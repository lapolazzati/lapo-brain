# AETHON — Technical Architecture (v0.1)

> **Design goal:** two architects + local operators, zero ops staff, zero secretaries. Every administrative task is either automated, agent-handled, or deleted. Humans approve; agents produce.

Companion to the Operating Manual. Lives at `/playbooks/technical-architecture.md` on Hetzner.

---

## 1. Architecture principles

1. **One source of truth per data type.** Sync problems come from the same fact living in two systems. Each type of data has exactly one home; everything else links to it.
2. **Agents write, humans approve.** No system requires manual data entry as its primary input. Agents draft CRM updates, Linear issues, digests, follow-ups; a human approves in one click or edits.
3. **Markdown + git underneath everything.** Knowledge is plain text in a git repo. Any editor works (Tolaria included), agents read/write it natively, nothing is locked in a SaaS.
4. **Self-host the brain, rent the edges.** Hetzner holds knowledge, agents, and dashboards. Commodity functions (email, CRM, PM) are rented SaaS chosen for API/MCP quality, because agent-friendliness is now a primary selection criterion.
5. **24-hour landing rule.** Slack and calls are ephemeral. Anything decided must land in its source of truth within 24h — and an agent does the landing, not a person.

## 3. Hetzner — server layout

Single VPS, Docker Compose, Caddy in front. Services:

```
hetzner/
  caddy            ← TLS + reverse proxy (warroom.<client>.aethon.x, cal.aethon.x, n8n.aethon.x)
  vault-mirror     ← bare git mirror of the GitHub repo (cron pull; agents read locally)
  skill-runner     ← Claude Code jobs: /marketresearch, /debrief, /warroom… (queue via n8n)
  agents/<client>  ← one Slack-socket agent per client, scoped to /clients/<name>/
  n8n              ← automation glue, webhooks
  cal              ← Cal.com (optional self-host)
  backups          ← restic → Hetzner Storage Box, nightly, tested monthly
```

Rules: Tailscale-only SSH; per-client agents have filesystem scoping to their own folder; every agent action appended to `agent-log.md`; secrets injected from 1Password, never in the repo.

## 4. Knowledge sync (the vault)

- **Remote:** private GitHub repo (`aethon/vault`). **Editors:** Tolaria or any markdown app on a local clone. **Agents:** read/write the Hetzner mirror, commit with `agent:` prefix.
- **Sync discipline:** humans pull before working sessions, commit at Friday vault-commit (and whenever a deliverable ships). Agents commit continuously. Merge conflicts are rare in practice because folders are owned: humans own `/playbooks` and `/corridors` prose; agents own `/clients/*/04-learnings` appends and `03-pipeline` mirrors.
- **GitHub Actions:** on push → regenerate HTML views of manuals, lint for missing `[source — date — confidence]` tags in `/01-research`, ping Slack if a client folder hasn't been touched in 7 days.
- New person setup: GitHub invite + clone + Tolaria pointed at the folder. Ten minutes, no IT.

## 5. CRM structure (Attio)

**Objects**

- `Company` — attributes: type (client / prospect / partner / platform / fund), corridor(s), sector, size, source, vault-link.
- `Person` — role (buyer / champion / operator / CM-candidate / investor / advisor), language, corridor, relationship-owner.
- `Engagement` (custom object) — client, corridor, sprint type, start date, retainer €, success-fee %, kill criteria (link to 00-brief), status.
- `Signal` (custom object) — company, trigger type (funding / intl hiring / localized site / partner news), date, confidence, intro angle. Written by `/signals`.

**Lists / pipelines**

- **Client pipeline:** Signal → Qualified → Diagnostic → Proposal → Active sprint → Install / Stop → Alumni (Expansion Board).
- **Partner pipeline:** Identified → Contacted → In conversation → Active corridor partner.
- **CM bench:** People where role = CM-candidate, grouped by corridor + sector. This list *is* the AETHON Pod product.
- **Fund relationships:** Companies where type = fund, with portfolio links — the warm-intro engine.

**Hygiene is agent work:** Attio auto-logs email/calendar; `/debrief` drafts the rest (stage moves, next steps, new contacts) for one-click approval. A human never types into a CRM field as a routine activity.

## 6. Linear structure — yes, Linear

Three teams, not one-per-client (clients are projects, not teams — teams are *kinds of work*):

| Team | Contains | Cycle |
|---|---|---|
| **DELIVERY** | One project per engagement (e.g. `Guday — Italy Validation`). Project milestones = the 5 Launch OS phases. | 1-week cycles |
| **BUILD** | Skills, agents, infra. One project per skill being built (`/marketresearch v1`). | 2-week cycles |
| **GROWTH** | AETHON's own pipeline, content from Atlas data, partnerships, hiring. | 1-week cycles |

**Conventions**

- Labels: corridor (`EU→BR`, `US→IT`…), `waiting-on-client`, `agent-draftable` (an agent can produce the first version), `senior-call` (architect only).
- Project templates: "Validation Sprint" and "Activation Sprint" templates pre-load the standard issues + milestones, so spinning up an engagement = one click after Attio hits "Active sprint" (n8n does it automatically).
- Definition of done for any client-facing issue includes "landed in vault/CRM" — the 24-hour rule enforced at the workflow level.

**Agents inside Linear (the no-secretary part)**

- `/debrief` creates issues from call commitments, assigned + labeled, linked to the Attio record.
- The **triage agent** reads new issues each morning: dedupes, labels, drafts anything `agent-draftable`, flags only true decisions to a human.
- The **reconciliation agent** (weekly) diffs Attio next-actions vs. open Linear issues and flags orphans in both directions — the sync watchdog.
- The **cycle agent** (Friday) drafts the weekly summary per project → feeds `/warroom`.

What humans do in Linear: move their own issues, write decisions in comments. That's it.

## 7. Sync architecture — who owns what

| Data type | Source of truth | Everyone else |
|---|---|---|
| Knowledge, research, learnings | **Vault (git)** | CRM + Linear link to vault notes |
| Pipeline & relationships | **Attio** | Linear projects link to Attio records |
| Tasks & execution state | **Linear** | Slack threads link to issues |
| Conversations | **Slack** (ephemeral) | Decisions must land elsewhere within 24h — agent-enforced |
| Meetings & transcripts | **Claap** | `/debrief` fans out to vault + Attio + Linear |
| Money (invoices, contracts) | **Accounting tool + Drive `/legal`** | Attio engagement links to contract |

**The standard flow (one meeting, zero admin):**

```
Claap transcript ready
  └→ n8n webhook → skill-runner: /debrief
       ├→ vault: /clients/<x>/04-learnings/2026-06-10-call.md  (summary, objections, learnings)
       ├→ Attio: stage/next-step draft → human one-click approve
       ├→ Linear: issues for each commitment, assigned, labeled
       └→ Slack: 5-line recap posted in the client channel
```

**The watchdog:** weekly reconciliation agent compares the three systems and posts a diff ("Attio says proposal promised to Conte; no Linear issue exists"). Sync is not a human chore — it's a monitored, automated property.

## 8. The zero-ops map

Every task a secretary or ops hire would do, and who does it instead:

| Admin task | Handled by |
|---|---|
| Scheduling & rescheduling | Cal.com links per engagement; agent proposes slots in Slack |
| Meeting notes & CRM entry | Claap + `/debrief` |
| Pre-meeting prep | `/brief`, auto-triggered 30 min before calendar events with external attendees |
| Status reports to clients | `/warroom` weekly digest, human-approved |
| Task creation & triage | `/debrief` + triage agent in Linear |
| Follow-up emails | Agent drafts in Gmail from debrief commitments; human sends |
| Inbox triage | Agent labels (client / prospect / partner / noise) + drafts replies for routine threads |
| Invoicing | Template invoices auto-drafted from Attio engagement terms on the 1st; accountant (external, few hrs/mo) handles filings |
| Proposal assembly | `/proposal` from template |
| Document formatting | Vault markdown → pandoc/pptx pipeline on Hetzner |
| Hiring funnel admin | Applications land in Attio (CM bench / operator list); agent screens against scorecard, drafts rejections, schedules interviews |
| Travel booking | Don't systematize at this size — book directly |

The only recurring human admin left: approving drafts, signing things, and the Friday vault commit.

## 9. Job boards

Two distinct uses — hiring channels, and **signal sources** (a company posting "Country Manager Brazil" is an AETHON prospect *before* it's anyone's employer).

**9.1 Hiring local BD operators**

- **Brazil:** LinkedIn (primary), Vagas.com, Catho, InfoJobs.com.br, 99Jobs; SDR/BD communities and inside-sales groups (the Receita Previsível ecosystem) often beat boards for quality; Workana for fractional starts.
- **Italy:** LinkedIn, InfoJobs.it, Indeed.it; university career services (Bocconi, Politecnico) for hungry juniors.
- **Remote / multi-market:** Wellfound (startup-native talent), Otta, Dynamite Jobs, RemoteOK.
- Always-on: a "Work with AETHON" page + the talent pitch from the Operating Manual; warm referrals from fund platform teams.

**9.2 CM-level talent**

Rarely found on boards. Sources: LinkedIn search (current/former country managers in corridor+sector), VC platform-team referrals, operators we've already worked with (the bench), and — best of all — people we *meet as prospects' hires*. Every strong person encountered in any engagement gets tagged `CM-candidate` in Attio.

**9.3 Boards as signal sources (feeds `/signals`)**

Monitor LinkedIn Jobs, Wellfound, and target-company career pages for: "Country Manager <X>", "Head of <country>", "International Sales", first hire with a local language requirement. n8n scrapes/polls weekly → `Signal` records in Attio with trigger = `intl hiring`. The job board is our best free intent database.

## 10. Money — billing, expenses, and the finance loop

Same philosophy: agents produce, humans approve, an external accountant signs. No finance hire, no finance software beyond the minimum. (Entity and tax structure — what gets billed through the US company vs. an Italian one, transfer pricing between them — is a commercialista + US CPA decision, not a tooling decision. Get that answered once, in writing, before the third client.)

**10.1 Banking & FX**

- **Mercury** for the US entity (USD operating account + cards).
- **Wise Business** for EUR/BRL/multi-currency receiving and cards (local account details in BRL matter for Brazilian clients).
- **FX policy (write it down, stop thinking about it):** invoice in the client's strong currency when possible. BRL receipts auto-convert to EUR/USD monthly above a threshold (e.g., R$20k). Never hold currency to speculate — we are not a treasury desk.

**10.2 Invoicing**

- **Italy:** e-invoicing via SDI is mandatory — use Fatture in Cloud (or whatever the commercialista runs). **US entity:** Stripe Invoicing or Mercury's native invoicing.
- **The agent loop:** on the 1st, n8n reads Attio `Engagement` terms (retainer, currency, entity) → drafts the invoice → human one-click approves → sent. Overdue chasing is agent-drafted reminder emails at +7/+14/+21 days; a human only gets involved at +21.

**10.3 Success-fee tracking (the custom part)**

The "retainer + % of revenue generated" model needs its own machinery, because nobody's software does this for you:

- Contract clause: client reports corridor revenue quarterly (simple template), with audit-on-request right.
- Attio `Engagement` gets fields: `success-fee %`, `attribution rule` (what counts as "revenue we generated" — define it in the contract, e.g., named accounts list), `revenue reported`, `fees invoiced`.
- Quarterly true-up workflow: agent sends the revenue-report request → logs the response → drafts the success-fee invoice → human approves. The named-accounts list in the CRM *is* the attribution evidence — another reason CRM hygiene is non-negotiable.

**10.4 Expenses**

- Business cards only (Mercury/Wise) — no reimbursement workflows, ever.
- Receipts: forward/photo to `expenses@` → n8n files it in Drive `/finance/<year>/<month>/` + appends a ledger row (date, vendor, amount, currency, engagement tag, card).
- Engagement tagging matters: travel and tools billed to a client per contract get tagged at capture, not reconstructed later.
- Monthly close: external accountant per entity (commercialista in IT, bookkeeper/CPA for US) works from the Drive folder + bank feeds. A few hours a month, fixed fee.

**10.5 The monthly money digest**

One agent job, first business day of the month, posted to a private founders channel: cash position across all accounts (normalized to EUR) · MRR from retainers · invoices outstanding + overdue · success fees accrued vs. invoiced · burn and months of runway · one-line anomalies ("Klaaryo invoice 12 days overdue"). If a number in that digest surprises you, the system failed — fix the capture, not the digest.

**10.6 Contracts & signing**

Drive `/legal/<client>/`, linked from the Attio engagement. E-signature via PandaDoc or Dropbox Sign — PandaDoc is worth considering since `/proposal` output can land there as a send-and-sign doc, closing the loop from skill → signed contract.

## 11. Security & access

- Tailscale for all server access; no public SSH.
- 1Password vaults: `core` (founders), `client-<name>` (scoped, shared with the operator on that engagement only).
- Per-client agent isolation: filesystem scope + separate Slack tokens + separate API keys, so a compromised channel never crosses clients.
- Data residency: vault and datasets on Hetzner (EU). LGPD/GDPR note in every client contract about what we process and where.
- Offboarding checklist in `/team`: GitHub, Slack, 1Password, Attio, Linear, Tailscale — six revocations, five minutes.

## 12. Build order

**Week 1 — the spine:** GitHub vault repo + folder structure; Attio workspace with the objects/pipelines above; Linear with the 3 teams + sprint templates; Slack Connect channel for the active client; Cal.com links.
**Week 2 — the loop:** Claap → n8n → `/debrief` flow end-to-end on real calls; `/brief` triggered from calendar; Attio "Active sprint" → Linear project automation.
**Week 3 — the showcase:** first client Slack agent live (read-only answers + weekly digest); `/warroom` page served from Hetzner; reconciliation watchdog on.
**Then:** `/marketresearch` v1 (wave 1 skill), `/signals` polling job boards, and only after that anything wave 3.

Rule of thumb throughout: **don't build the tool until the manual version has run twice.**
